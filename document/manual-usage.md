# 使用手順書（人間向け）

対象読者：この html5-skeleton を使って静的サイトを作る人（エンジニア以外も想定）。
コマンドの意味よりも「何をどこに置けば何が起きるか」を優先して説明する。

---

## 0. 前提

- Node.js が入っていること（バージョン指定なし。手元に無ければ公式サイトから入れる）。
- ターミナルでこのリポジトリのルート（`package.json` がある場所）に移動して以下を実行する。

```bash
npm install
```

これで `node_modules/` に必要なツール一式が入る。以降のコマンドはすべてこのルートで実行する。

---

## 1. まず動かす

```bash
npm start
```

これだけで良い。実行すると、

1. 一度フルビルドが走る（`dist_uncompressed/` と `dist/` の両方が作られる）
2. そのままファイル監視状態に入り、`src/` を編集するたびに自動で再ビルドされる
3. ブラウザが自動で開き、`http://localhost:3000` 相当のアドレスでプレビューできる（BrowserSync）
4. 保存するとブラウザも自動リロードされる

止めるときはターミナルで `Ctrl + C`。

途中で `dist/` や `dist_uncompressed/` の中身を手で書き換えてはいけない。ビルドのたびに上書きされて消える。編集対象は常に `src/` 側。

---

## 2. ページを1枚増やしたいとき

やることは2つ、セットで行う。

1. `src/htdocs/` の下に新しい `.ejs` ファイルを置く
   - 例：`src/htdocs/about/index.ejs`
   - ファイル名の先頭に `_`（アンダースコア）を付けない。付けると「他ファイルから読み込まれる部品」扱いになり、単独ページとしてビルドされない。
2. `src/content/text/pages/` の下に、1と同じ相対パスで `.json` を置く
   - 上の例なら `src/content/text/pages/about/index.json`
   - 中身はページタイトルなどの最小限のメタ情報（既存の他ページの `.json` を1つ開いてコピーし、値だけ書き換えるのが早い）

置いたら **`npm start` をやり直す**（再起動が必要）。content JSON は起動時に一度だけ読み込まれる仕組みなので、`npm start` を動かしたままファイルを追加しても認識されない。

サイト全体の名前やURLなど共通データは `src/content/text/site.json` にある。ここも編集後は再起動が必要。

---

## 3. スタイル（CSS）やスクリプト（JS）を1枚増やしたいとき

- `src/htdocs/` の下に `.scss` または `.js`（`.ts` も可）を置くだけで良い。設定ファイルへの登録は不要。
- ファイル名の先頭に `_` を付けたものは「他から読み込まれる部品」（例：`_reset.scss`）。単独では出力されない。
- `_` を付けないもの（例：`base.scss`、`common.scss`）は、それ単体で1つの出力になる。

こちらは `npm start` を再起動しなくても、保存すれば自動で再ビルド・反映される（ページ追加とは違い、再起動不要）。

---

## 4. 画像を使いたいとき

用途によって置き場所が2通りある。

### 4-1. CSSやJSに埋め込みたい画像（通常のアイコンや小さい画像など）

`src/content/img/`（または `src/htdocs/assets/img/` などSCSS/JSから相対パスで参照できる場所）に置き、SCSS/JS側から通常どおり相対パスで参照する。ビルド時に自動でインライン化される。

### 4-2. サイズの大きい写真をWebPに変換したいとき

プロジェクトルート直下の `img2webp/` フォルダに `.jpg` / `.png` / `.jpeg` を置く。`npm start` を動かしていれば自動的に `.webp` に変換され、**同じ `img2webp/` フォルダ内に**変換済みファイルが書き戻される。`dist/` には出力されないので注意（別系統の仕組み）。

`npm start` を使わず変換だけしたい場合は次のコマンドでも良い。

```bash
npm run webp
```

画質は固定で90（設定ファイル `config/image-optimization.config.js` を参照。通常は触らなくてよい）。

---

## 5. 公開前のチェック

### HTMLの構文チェック

```bash
npm run validate:html
```

`dist_uncompressed/htdocs/` 配下のHTMLをチェックする。エラーが出た場合はファイル名・行番号が表示されるので、対応する `src/htdocs/**/*.ejs` を直す。

### TypeScriptの型チェック（`.ts` ファイルを使っている場合のみ）

```bash
npm run type-check
```

**重要**：`npm start` のビルドは型エラーがあっても素通りしてしまう（型情報を捨てて変換するだけのため）。`.ts` を書いたら、公開前に必ずこのコマンドで型エラーの有無を確認する。

---

## 6. 公開用ファイルの場所

`npm start` を一度でも実行すれば、以下ができあがっている。

- `dist/` … **本番公開用**。圧縮済み。これをWebサーバーにアップロードする。
- `dist_uncompressed/` … 差分確認・デバッグ用。圧縮していないので中身が読みやすい。公開しない。

---

## 7. アップロード（デプロイ）

このリポジトリには自動デプロイの npm コマンドは無い。VSCode の SFTP 拡張機能（設定ファイル：`.vscode/sftp.json`）を使う想定になっている。

- アップロード対象は `dist/` のみ（`context: "dist"` と設定済み）。
- `staging` / `production` の2プロファイルがあるが、ホスト名・ユーザー名・パスワードは空のプレースホルダーなので、案件ごとに手元の環境で埋める必要がある（この値をGit管理下に置いたままコミットしないよう注意）。
- `staging` は保存時に自動アップロードする設定、`production` は手動アップロードのみの設定になっている。うっかり `production` に自動反映されることは無いが、`staging` は保存するたびにアップロードされる点に注意。

---

## 8. コード品質チェック（任意）

- ESLint：`npx eslint src`
- Stylelint：`npm start` に組み込み済みだが普段はオフ。有効にしたいときだけ次のように起動する。

```bash
STYLELINT=1 npm start
```

有効にすると、保存のたびにSCSSを自動整形（自動修正）しながら再ビルドする。通常の作業では毎回SCSSが書き換わって煩わしいため、必要なとき限定で使う。

---

## 9. よくあるつまずき

| 症状 | 原因・対処 |
| --- | --- |
| ページを追加したのにブラウザに出てこない | `npm start` の再起動を忘れている（content JSONは起動時にしか読まれない） |
| `img2webp/` に画像を置いたのに変換されない | `npm start`（または `npm run webp`）を動かしていない。動かしていても `.webp` 拡張子のファイルは監視対象外にしているので、元の `.jpg`/`.png` を置いたか確認する |
| `dist/` を直接編集したのに次のビルドで消えた | 想定どおりの動作。`dist/` は生成物なので編集対象は常に `src/` |
| 新しいSCSS/JSファイルがビルドに含まれない | ファイル名の先頭が `_` になっていないか確認する（`_` 付きは部品扱いで単独出力されない） |
| 型エラーに気づかず公開してしまった | `npm start` は型チェックをしないため。`.ts` を使うプロジェクトでは公開前に必ず `npm run type-check` を実行する |
